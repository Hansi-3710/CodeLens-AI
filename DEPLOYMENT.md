# Deployment

## Go live in ~15 minutes — exact steps, in order

This is the literal sequence. Each step only needs what the previous step
produced — no guessing, no placeholders left in a config file for you to
remember to edit later.

### 0. Push to GitHub

```bash
cd llm-code-intelligence
git init
git add .
git commit -m "Initial commit"
```
Then create an empty repo on GitHub (github.com/new — no README/gitignore,
you already have those) and:
```bash
git remote add origin https://github.com/<your-username>/<your-repo>.git
git branch -M main
git push -u origin main
```

### 1. Deploy the backend to Render *first* (you need its URL for step 2)

1. [render.com/deploy](https://render.com) → New → Blueprint → connect the
   GitHub repo you just pushed. Render auto-detects `render.yaml` (already
   in the repo) and provisions the API service + a managed Postgres
   database together.
2. In the Render dashboard, set these two service env vars (the blueprint
   marks them `sync: false` deliberately — secrets aren't meant to live in
   a committed file):
   - `OPENAI_API_KEY`
   - `HUGGINGFACE_API_TOKEN`
3. `SECRET_KEY` is auto-generated by Render (`generateValue: true` in the
   blueprint) and `DATABASE_URL` is wired automatically from the Postgres
   instance the blueprint also creates — nothing to do for either.
4. Set `ENVIRONMENT=production`. **This matters**: `app/config.py` now
   refuses to start in production with a default `SECRET_KEY`, a
   local-looking `DATABASE_URL`, or a `localhost` CORS origin — so if
   something above was missed, the service fails at boot with a clear
   error naming exactly what's wrong, instead of starting broken.
5. Wait for the first deploy to finish, then copy the service's URL
   (`https://<something>.onrender.com`).
6. Run the migration and seed the model registry once, via Render's
   dashboard → Shell on the running service:
   ```bash
   alembic upgrade head
   python scripts/seed.py
   ```

### 2. Deploy the frontend to Vercel

1. [vercel.com/new](https://vercel.com) → import the same GitHub repo,
   root directory `frontend/`. `vercel.json` (already in the repo) sets
   the build command and framework preset.
2. In Vercel's project settings → Environment Variables, add:
   - `VITE_API_BASE_URL` = the Render URL from step 1.5, e.g.
     `https://llm-code-intel-api.onrender.com`
3. Deploy. Copy the resulting Vercel URL
   (`https://<something>.vercel.app`).

### 3. Close the loop: tell the backend about the frontend's URL

Back in Render's dashboard, set one more env var on the backend service:
- `ALLOWED_ORIGINS` = the Vercel URL from step 2.3, e.g.
  `https://your-app.vercel.app`

(Plain string, no brackets or quotes needed — `app/config.py` parses a
comma-separated value directly; also accepts a comma-separated list if you
have multiple frontend URLs, e.g. a preview + production Vercel domain.)

Redeploy the backend so the new env var takes effect.

### 4. Done

Anyone can now open the Vercel URL, register an account, and use the
platform — no local setup, no cloning the repo. That's what "pushed to
GitHub and live" means here: two connected dashboards (Render, Vercel)
watching the same repo, redeploying automatically on every future push to
`main`.

**One real limitation, stated plainly:** `POST /solutions/{id}/execute`
(running generated code in the Docker sandbox) needs a reachable Docker
daemon, which Render's standard web service does not provide — see
"Execution sandbox in production" below before expecting that one specific
feature to work on this exact setup. Everything else (auth, creating
experiments, generating solutions from real models, the leaderboard,
analytics, similarity, clustering) works on the deployment above with no
further changes.

## Reference: what each piece does

The same `backend/Dockerfile` works on any container host (Fly.io, AWS App
Runner, ECS, a plain VM running `docker compose`) if Render isn't the right
fit — nothing in the app code is Render-specific.

### Execution sandbox in production

`app/execution/docker_worker.py` needs a **reachable Docker daemon** to run
generated code — this is the one piece of infrastructure that doesn't fit a
typical serverless/PaaS web service, which usually can't run
docker-in-docker. Two supported paths:

1. **A VM/dedicated host** (EC2, a Droplet, a Render "private service" with
   Docker access) — mount `/var/run/docker.sock` exactly as
   `docker-compose.yml` already does for local dev. This is the simplest
   path and what the current `render.yaml` assumes if you switch its
   service `runtime` to a Docker-socket-accessible host type.
2. **A separate execution worker** — split `execution/` into its own
   service (a small VM or a container-capable job runner like AWS Fargate
   with Docker-outside-of-Docker) that the API calls over an internal queue
   instead of invoking `docker_worker.py` in-process. This is the more
   "correct" production architecture (keeps untrusted-code execution off
   the same host as the API) but is a larger change than Phase 9 scope —
   flagged here as the next architectural step, not implemented.

Until either is set up, `/solutions/{id}/execute` will return a sandbox
infrastructure error (see `DockerException` handling in `docker_worker.py`)
rather than silently failing — the failure mode is explicit.

## Database → managed Postgres

`render.yaml` provisions this automatically. If hosting the DB elsewhere
(RDS, Supabase, Neon), just point `DATABASE_URL` at it — nothing else in the
app assumes Render specifically.

## Secrets checklist

| Secret | Where it's used | Required for |
|---|---|---|
| `SECRET_KEY` | `app/core/security.py` (JWT signing) | Auth to work at all |
| `DATABASE_URL` | `app/database/session.py` | Everything |
| `ALLOWED_ORIGINS` | `app/main.py` (CORS) | The deployed frontend's requests to be accepted at all |
| `OPENAI_API_KEY` | `app/llm/providers/openai_provider.py` | GPT-family generation |
| `HUGGINGFACE_API_TOKEN` | `app/llm/providers/huggingface_provider.py` | Llama/Gemma-family generation |

Never commit real values — `backend/.env.example` and `frontend/.env.example`
document the shape, `.env` is git-ignored. `app/config.py`'s
`validate_for_production()` refuses to boot with `ENVIRONMENT=production`
if `SECRET_KEY`, `DATABASE_URL`, or `ALLOWED_ORIGINS` are still their
insecure local defaults — tested in `tests/test_production_config_guard.py`.

## What's verified vs. what needs real infrastructure to confirm

This project was built and hardened in a sandbox with **no Docker daemon,
no Postgres server, no browser, and no network egress to
openai.com/huggingface.co** — so here's exactly what that means for
confidence level:

| Verified in this environment | Needs real infra to confirm |
|---|---|
| FastAPI app boots, all routes wire correctly, OpenAPI schema generates cleanly (14 paths, 26 schemas) | Real OpenAI/HuggingFace API calls (provider code is written correctly against each SDK's documented interface, but untested against live endpoints) |
| Full auth flow, CRUD, ownership isolation, rate limiting (proven via real repeated requests), N+1 query elimination (proven via query counting), 98 backend tests / 91% coverage | Real Docker sandbox execution (call *contract* — network-disabled, memory-capped, no capabilities, read-only root, pids-limited — is unit-tested with a mocked client; `test_docker_worker.py`'s `requires_docker` integration test is written and auto-skips here, but will run for real wherever Docker is available) |
| Alembic migrations apply cleanly (tested against SQLite as a stand-in dialect, including a hand-corrected index migration) | The same migrations against real Postgres (types are Postgres-specific — `postgresql.UUID` — so this should be a formality, but hasn't been run against a live Postgres instance; CI does apply them to a real Postgres service container on every push, which is the closest verification available without live infra) |
| All ML/analysis logic (AST, complexity, style, similarity, PCA, UMAP, clustering, cross-validated classification, Pearson+Spearman statistics) — real computation, real assertions | Real Sentence-Transformer embeddings (download requires network access to huggingface.co; `embed()`'s logic is correct and unit-tested with a mocked model, but never run against real weights) |
| Frontend builds, typechecks, and passes its Jest suite cleanly; the production `VITE_API_BASE_URL` build-time substitution is proven to land in the actual output bundle | A real browser session against a deployed instance — no browser is available in this environment, so nothing here has been clicked through manually |
| The full request/response cycle for every endpoint, exercised via HTTP through FastAPI's real ASGI stack (not just function calls) | The same cycle across two separately-deployed services (Render + Vercel) talking over real CORS, not FastAPI's in-process test transport |

None of this changes the code — it's the honest boundary of what "tested"
means when the sandbox itself can't reach Docker Hub, a Postgres port, a
browser, or the outside internet. Once deployed via the runbook above, the
one meaningfully untested path end-to-end is the Docker sandbox (needs its
own infrastructure decision — see above) and the two live LLM providers
(need real API keys, which this environment was never given).

## Known non-issue: `npm audit` finding

`npm audit` reports one moderate-severity vulnerability in `esbuild` (via
Vite's dev server) — it allows a malicious website to send requests to
`npm run dev`'s local dev server and read the response. This does not
affect the deployed production build: Vercel serves a static file bundle
(`npm run build`'s output) with no dev server running at all, so there's
nothing for this vulnerability to reach. Left unfixed rather than pulling
in Vite 8 (a breaking change) this close to shipping; safe to address
later with `npm audit fix --force` if desired.
